## 🎯 Migration Naming Conventions

### Option 1: Sequential Numbers (Simple) ✅ What You Have
```
001_initial_schema.sql
002_add_competitions_and_scoring.sql
003_add_notifications.sql
004_add_user_avatars.sql
```

**Pros:** Easy to understand, clean
**Cons:** Team conflicts if two people create same number

### Option 2: Timestamps (Supabase Default)
```
20240630120000_initial_schema.sql
20240630130000_add_competitions.sql
20240701090000_add_notifications.sql
```

**Format:** `YYYYMMDDHHMMSS_description.sql`

**Pros:** No conflicts, automatic ordering
**Cons:** Harder to read

**Generate with CLI:**
```bash
supabase migration new add_feature_name
# Creates: 20240630120000_add_feature_name.sql
```

### Option 3: Hybrid (Recommended for Teams)
```
20240630_001_initial_schema.sql
20240630_002_add_competitions.sql
20240701_003_add_notifications.sql
```

**Pros:** Date context + readable numbers
**Cons:** Requires manual coordination

## 📝 Migration File Structure

Each migration file should be:

### Simple Migration
```sql
-- Migration: 003_add_feature
-- Description: Add notifications table

CREATE TABLE notifications (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  user_id UUID REFERENCES auth.users(id),
  message TEXT,
  created_at TIMESTAMPTZ DEFAULT NOW()
);

-- Add RLS policies
ALTER TABLE notifications ENABLE ROW LEVEL SECURITY;

CREATE POLICY "Users can view own notifications"
  ON notifications FOR SELECT
  USING (auth.uid() = user_id);
```

### Complex Migration with Rollback
```sql
-- Migration: 003_add_feature
-- Up Migration (apply changes)

-- Create table
CREATE TABLE notifications (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  user_id UUID REFERENCES auth.users(id),
  message TEXT,
  created_at TIMESTAMPTZ DEFAULT NOW()
);

-- Rollback (separate file: 003_add_feature.down.sql)
-- Down Migration (undo changes)
DROP TABLE IF EXISTS notifications;
```

## 🚀 How Supabase Reads Your Folder

### GitHub Integration
1. You push code to GitHub
2. Supabase webhook detects changes in `supabase/migrations/`
3. Scans for new `.sql` files
4. Applies them in order (alphanumeric sort)
5. Tracks applied migrations in `supabase_migrations` table

### Supabase CLI
```bash
# Apply all pending migrations
supabase db push

# Create new migration
supabase migration new feature_name

# List migrations status
supabase migration list
```

## 🎯 Migration Best Practices

### 1. One Change Per Migration
❌ **Don't:**
```sql
-- 003_big_update.sql
CREATE TABLE notifications (...);
ALTER TABLE leagues ADD COLUMN (...);
CREATE TABLE messages (...);
```

✅ **Do:**
```sql
-- 003_add_notifications.sql
CREATE TABLE notifications (...);

-- 004_update_leagues.sql
ALTER TABLE leagues ADD COLUMN (...);

-- 005_add_messages.sql
CREATE TABLE messages (...);
```

### 2. Never Edit Applied Migrations
Once a migration is applied to any database, **never edit it**.

❌ **Don't:**
```sql
-- Editing 001_initial_schema.sql after it's applied
```

✅ **Do:**
```sql
-- Create new 006_fix_initial_schema.sql
ALTER TABLE leagues ADD COLUMN missing_field TEXT;
```

### 3. Test Before Pushing
```bash
# Test locally first (if using CLI)
supabase start
supabase migration up
# Test your app
supabase migration down  # Rollback if needed
```

### 4. Use Transactions
```sql
-- Wrap related changes in transaction
BEGIN;

  CREATE TABLE notifications (...);

  ALTER TABLE users ADD COLUMN notify_enabled BOOLEAN DEFAULT true;

  CREATE INDEX idx_notifications_user ON notifications(user_id);

COMMIT;
```

## 🆘 Common Issues

### Issue: "Migration already applied"
**Solution:** Check `supabase_migrations` table in your database
```sql
SELECT * FROM supabase_migrations ORDER BY version;
```

### Issue: Migrations out of order
**Solution:** Use timestamps or coordinate with team
```bash
# Check what's applied
SELECT version, name FROM supabase_migrations;
```

### Issue: Failed migration
**Solution:**
1. Check Supabase logs
2. Fix SQL error
3. Create new migration with fix
4. Never edit the failed migration
